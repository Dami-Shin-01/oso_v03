# 문서 명명 규칙 (Document Naming Convention)

**목적**: 프로젝트 문서의 위계와 역할을 명확히 하여 관리 및 탐색을 용이하게 함

**적용일**: 2025-10-22
**버전**: 1.0

---

## 📋 명명 규칙

### 기본 형식

```
{번호}-{타입}-{내용}.md

예시:
01-PLAN.md
10-REF-DESIGN-SYSTEM.md
20-REPORT-CSS-EXTRACTION.md
```

### 구성 요소

1. **번호 (2자리)**: 문서의 계층 및 순서
   - 00-09: 메인 문서
   - 10-19: 참조 문서
   - 20-29: 분석 리포트
   - 30-39: 데이터 문서
   - 90-99: 메타 문서

2. **타입 (선택)**: 문서의 역할 (참조/리포트 문서에 사용)
   - `REF-`: 참조 문서 (Reference)
   - `REPORT-`: 분석 리포트 (Report)
   - `DATA-`: 데이터 문서 (Data)

3. **내용**: 문서의 실제 내용을 나타내는 설명
   - 대문자 사용
   - 하이픈(-)으로 단어 구분
   - 명확하고 간결하게

## 📚 계층 구조 (4단계)

### Level 0: 메인 문서 (00-09)
**목적**: 프로젝트의 핵심 문서

| 번호 | 파일명 | 설명 | 필수 |
|------|--------|------|------|
| 00 | 00-README.md | 프로젝트 진입점 | ✅ 필수 |
| 01 | 01-PLAN.md | 전체 프로젝트 계획 | ✅ 필수 |
| 02 | 02-STATUS.md | 진행 상황 요약 | ✅ 필수 |
| 03 | 03-ROADMAP.md | 로드맵 (향후) | 선택 |
| 04-09 | (예약) | 향후 확장 | - |

**특징**:
- 가장 자주 참조되는 문서
- 프로젝트 전체를 이해하는 데 필수
- 타입 prefix 없이 간결하게

### Level 1: 참조 문서 (10-19)
**목적**: 기술 스펙, 가이드, 레퍼런스

| 번호 | 파일명 | 설명 | 필수 |
|------|--------|------|------|
| 10 | 10-REF-DESIGN-SYSTEM.md | 디자인 시스템 | ✅ 필수 |
| 11 | 11-REF-LIBRARY-VERSIONS.md | 라이브러리 버전 | ✅ 필수 |
| 12 | 12-REF-API-DOCS.md | API 문서 (향후) | 선택 |
| 13 | 13-REF-COMPONENT-GUIDE.md | 컴포넌트 가이드 (향후) | 선택 |
| 14-19 | (예약) | 향후 확장 | - |

**특징**:
- 개발 중 자주 참조
- 기술적 세부사항 포함
- `REF-` prefix 사용

### Level 2: 분석 리포트 (20-29)
**목적**: 분석 과정 및 결과 문서

| 번호 | 파일명 | 설명 | 필수 |
|------|--------|------|------|
| 20 | 20-REPORT-CSS-EXTRACTION.md | CSS 추출 보고서 | ✅ 필수 |
| 21 | 21-REPORT-PERFORMANCE.md | 성능 분석 (향후) | 선택 |
| 22 | 22-REPORT-SEO-AUDIT.md | SEO 분석 (향후) | 선택 |
| 23 | 23-REPORT-ACCESSIBILITY.md | 접근성 분석 (향후) | 선택 |
| 24-29 | (예약) | 향후 확장 | - |

**특징**:
- 일회성 분석 결과
- 참고용 아카이브
- `REPORT-` prefix 사용

### Level 3: 데이터 문서 (30-39)
**목적**: 구조화된 데이터의 문서화 (향후)

| 번호 | 파일명 | 설명 | 필수 |
|------|--------|------|------|
| 30 | 30-DATA-ROOMS.md | 객실 데이터 문서 | 선택 |
| 31 | 31-DATA-FACILITIES.md | 부대시설 데이터 문서 | 선택 |
| 32-39 | (예약) | 향후 확장 | - |

**특징**:
- JSON 데이터의 설명 및 스키마
- 데이터 관리 가이드
- `DATA-` prefix 사용

### Level 9: 메타 문서 (90-99)
**목적**: 프로젝트 관리 문서

| 번호 | 파일명 | 설명 | 필수 |
|------|--------|------|------|
| 99 | 99-NAMING-CONVENTION.md | 명명 규칙 (이 문서) | ✅ 필수 |
| 98 | 98-CHANGELOG.md | 변경 이력 (향후) | 선택 |
| 97 | 97-CONTRIBUTING.md | 기여 가이드 (향후) | 선택 |
| 90-96 | (예약) | 향후 확장 | - |

**특징**:
- 프로젝트 관리용
- 메타데이터 및 규칙
- 가장 낮은 우선순위

## ✅ 명명 규칙 체크리스트

새 문서를 만들 때 다음을 확인하세요:

- [ ] **번호 범위 확인**: 문서 유형에 맞는 번호 범위 선택
- [ ] **중복 확인**: 같은 번호가 이미 사용 중인지 확인
- [ ] **타입 prefix**: 참조/리포트/데이터 문서는 prefix 추가
- [ ] **내용 명확성**: 파일명만으로 내용을 추측할 수 있는지 확인
- [ ] **일관성**: 기존 문서와 스타일이 일치하는지 확인

## 📖 예시

### ✅ 좋은 예시

```
00-README.md                        # 진입점, 간결함
01-PLAN.md                          # 메인 계획, 명확함
10-REF-DESIGN-SYSTEM.md            # 참조 문서, prefix 있음
20-REPORT-CSS-EXTRACTION.md        # 리포트, 내용 명확
```

### ❌ 나쁜 예시

```
README.md                           # 번호 없음 (정렬 불가)
plan-benchmark.md                   # 번호 없음, 소문자
REF-DESIGN.md                       # 번호 없음
15-design-system.md                 # 잘못된 번호 범위 (15는 참조 문서용)
REPORT-CSS.md                       # 번호 없음
```

## 🎯 명명 규칙의 장점

### 1. **명확한 위계**
- 번호만 보고 문서의 중요도와 역할을 즉시 파악
- 파일 탐색기에서 자동으로 논리적 순서 정렬

### 2. **확장성**
- 10단위로 그룹핑하여 향후 문서 추가 시 유연함
- 각 카테고리마다 10개씩 문서 추가 가능

### 3. **일관성**
- 모든 팀원이 동일한 규칙 적용
- 새 문서 작성 시 혼란 최소화

### 4. **검색 용이성**
- 번호로 빠른 검색 (예: "10-*" → 모든 참조 문서)
- 타입 prefix로 문서 유형 필터링 가능

## 📝 문서 작성 가이드

### 메인 문서 (00-09)
- **대상 독자**: 모든 프로젝트 참여자
- **톤**: 명확하고 간결하게
- **길이**: 중간~긴 문서 (200-600줄)
- **업데이트**: 자주 (프로젝트 진행에 따라)

### 참조 문서 (10-19)
- **대상 독자**: 개발자
- **톤**: 기술적이고 상세하게
- **길이**: 중간~긴 문서 (100-500줄)
- **업데이트**: 필요 시 (기술 변경 시)

### 리포트 (20-29)
- **대상 독자**: 분석 담당자, 의사결정자
- **톤**: 객관적이고 사실 기반
- **길이**: 짧음~중간 (50-300줄)
- **업데이트**: 거의 없음 (아카이브)

### 메타 문서 (90-99)
- **대상 독자**: 프로젝트 관리자
- **톤**: 공식적이고 규범적
- **길이**: 짧음~중간 (50-200줄)
- **업데이트**: 필요 시 (규칙 변경 시)

## 🔧 도구 및 자동화

### Git Hooks (향후)
```bash
# pre-commit hook: 파일명 검증
#!/bin/bash
for file in *.md; do
  if ! [[ $file =~ ^[0-9]{2}(-[A-Z-]+)?\.md$ ]]; then
    echo "Error: $file does not follow naming convention"
    exit 1
  fi
done
```

### VS Code Settings (향후)
```json
{
  "files.exclude": {
    "**/[0-9][0-9]-*.md": false
  },
  "search.exclude": {
    "**/9[0-9]-*.md": true
  }
}
```
